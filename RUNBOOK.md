# Operational Runbook

This runbook explains how a trade support analyst would investigate exception reports generated by this simulated project.

> All records in this project are simulated and created for portfolio demonstration only. No real trading, broker, client, OMS, EMS, or production data is used.

## Daily Control Flow

1. Receive raw FIX execution-report messages from `data/raw/fix_messages.txt`.
2. Parse FIX tag-value messages into `data/generated/fix_execution_reports.csv`.
3. Load internal orders, parsed FIX reports, broker drop-copy data, and expected order states into SQLite.
4. Run SQL reconciliation controls.
5. Export exception reports into `reports/exceptions/`.
6. Review exceptions by severity and document next action.

## How to Run

```bash
python -m pip install -r requirements.txt
make run
```

Equivalent manual sequence:

```bash
python -m fixops.generate_sample_data
```

## Exception Triage Guide

### Missing execution report
Meaning: an order exists internally, but no FIX execution report was received.

Likely checks:
- Confirm whether the order was actually routed.
- Check FIX session sequence gaps or disconnects.
- Check whether a reject was sent to another session.
- Escalate to connectivity/OMS support if the order is missing from broker acknowledgement flow.

### Missing broker drop copy
Meaning: FIX shows an execution, but broker drop-copy data does not contain a matching `ExecID`.

Likely checks:
- Confirm whether broker drop copy was delayed.
- Compare `ExecID`, `ClOrdID`, `OrderID`, symbol, side, quantity, and price.
- Escalate to broker support if drop-copy feed is incomplete.

### Rejected order
Meaning: the broker rejected the order.

Likely checks:
- Review tag `58=Text` for reject reason.
- Check price bands, symbol eligibility, session permissions, or quantity limits.
- Confirm whether the strategy retried, canceled, or left residual exposure.

### Stale open order
Meaning: latest order state is New or PartiallyFilled with leaves quantity remaining for more than the configured threshold.

Likely checks:
- Confirm whether the order should still be live.
- Check market status, limit price, and routing destination.
- Ask whether cancel/replace or manual cancel is needed.

### Quantity / price / side / symbol mismatch
Meaning: fields differ across internal order records, FIX reports, or broker drop-copy records.

Likely checks:
- Compare raw FIX message, parsed CSV, broker file, and internal record.
- Determine whether the mismatch is a parsing issue, source-system issue, or true broker discrepancy.
- Escalate only after confirming the source of truth.

### Duplicate ExecID
Meaning: same execution identifier appears more than once in the FIX stream.

Likely checks:
- Identify whether the duplicate is a replay, resend, or duplicate booking risk.
- Check sequence number and replay flags in a real system.
- Ensure downstream allocation/trade booking logic is idempotent.

### Unexpected fill
Meaning: a fill exists for a `ClOrdID` absent from internal order records.

Likely checks:
- Confirm whether internal order capture missed a routed order.
- Check if the order belongs to another account/session.
- Escalate immediately because unexpected fills can create position and P&L risk.

### CumQty / LeavesQty inconsistency
Meaning: `CumQty + LeavesQty` does not equal `OrderQty`.

Likely checks:
- Confirm whether the FIX report is malformed or parser coerced values incorrectly.
- Compare against broker drop copy and expected final state.
- Flag as high-priority data integrity issue.

## Severity Guide

| Severity | Exception Types | Why It Matters |
|---|---|---|
| High | unexpected_fill, duplicate_exec_id, cumqty_leavesqty_inconsistency, side_mismatch, symbol_mismatch | Can create incorrect position, duplicate trade, or material booking risk |
| Medium | missing_execution_report, missing_broker_drop_copy, quantity_mismatch, price_mismatch, order_status_mismatch | Can indicate incomplete lifecycle, stale source data, or downstream reconciliation break |
| Low/Medium | rejected_order, stale_open_order | Often expected in trading operations but requires review |

## Production-Support Notes

In a real environment, this workflow would include alerting, SLA thresholds, sequence-gap handling, idempotent processing, monitoring dashboards, broker escalation contacts, and audit trails. This project intentionally uses CSV, SQLite, and Python so the logic remains transparent and easy to discuss in interviews.
